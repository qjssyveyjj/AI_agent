## 二、各模块详细说明

### 1. 多模态交互前端

- **功能**：类似 ChatGPT 的聊天框，支持文本输入、粘贴剪贴板截图或上传图片文件。
- **技术选型**：Vue3 / React，使用`vuedraggable`或`react-dropzone`等组件。
- **通信方式**：若需流式输出，建议通过 **WebSocket** 或 **SSE** 连接后端；否则普通 HTTP 请求亦可。

### 2. 后端 API 网关与 Agent 调度器（核心）

- **框架**：Python **FastAPI**（异步性能好，生态丰富）。
- **职责**：
  - 接收前端消息（文本 + 图片），对图片做压缩/格式转换（如统一为 JPEG，限制分辨率）。
  - 管理会话上下文（对话历史，包含多模态消息），保存到 Redis 或数据库中。
  - 实现 **Agent 循环**：调用大模型 → 解析模型返回的“工具调用”指令 → 执行对应的内部 API → 将结果反馈给大模型 → 生成最终回复。
  - 内部 HTTP 客户端：使用 `httpx` / `aiohttp` 异步调用 `192.168.126.200/dev-api` 的各种端点，并处理认证（Token、IP 白名单等）。

### 3. 多模态大模型服务（私有化部署）

- **为什么需要多模态**：用户会上传截图，模型必须能“看懂”图片中的文字、界面元素、数据，才能提取关键信息（如订单号、错误提示）并结合内部 API 进行查询。
- **推荐模型**（按需选择，需支持**工具调用/函数调用**能力）：
  - **Qwen3-VL-Plus** （阿里通义千问视觉模型，原生支持 Function Call，中文理解优秀）
- **接口封装**：让模型服务提供 **OpenAI 兼容的 API**（`/v1/chat/completions`），方便与 LangChain 等框架集成，且支持 `tools` 参数定义。

### 4. 内部 API 工具集成层

将 `http://192.168.126.200/dev-api` 下的各个端点封装成**符合大模型 Function Calling 规范的工具描述**。例如：

json

```
{
  "name": "query_order_status",
  "description": "根据订单号查询订单状态及详细信息",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "订单号，一般为 ORD 开头"
      }
    },
    "required": ["order_id"]
  }
}
```



工具的实现就是一个 Python 异步函数，内部调用内部 API 并返回结构化数据。
可以维护一个**工具注册表**，Agent 启动时将其全部注入给大模型。

### 5. Agent 推理循环

当用户发来“截图 + 问题”时：

1. **用户消息构造**：将图片（base64 或 URL）和文本组成多模态消息体，连同历史消息一起发给大模型。
2. **模型思考**：多模态模型识别截图内容（如看到一个报错页面、一张表格），理解用户意图，决定是否需要调用工具。
3. **工具调用**：若模型返回 `tool_calls`（例如要求查询某个订单），后端解析参数，调用内部 API 得到结果。
4. **结果注入**：将工具返回的数据以“工具角色消息”形式追加到对话中，再次请求模型生成最终回答。
5. **循环**：若需要多步查询（先查用户信息，再查订单），可重复步骤 3-4，直到模型生成面向用户的自然语言回复。

**可选框架**：可直接使用 **LangChain** 的 `AgentExecutor` + `ChatOpenAI`（对接本地模型），或更灵活的 **LangGraph** 自定义状态图；也可完全手写控制循环，逻辑并不复杂。

### 6. 安全与权限

- 整个系统部署在**内网**，前端至后端可仅用 HTTP + JWT 或企业统一认证。
- 内部 API 的调用必须通过后端代理，绝不暴露给前端，避免内部接口被绕过。
- 模型自身无权限直接访问内部 API，所有工具执行都在后端可控范围内。
- 对敏感数据（如 API 返回的用户手机号）可在工具层进行脱敏后再送模型。

## 三、典型交互流程示例

**用户操作**：截取一张 ERP 系统的订单详情页，并输入“帮我查一下这个订单的最新物流状态”。

1. 前端将截图和文本发送到 FastAPI。
2. 后端构造多模态请求发送给 Qwen3-VL-Plus 模型，模型从截图中识别出订单号 `ORD20260623-001`。
3. 模型产生工具调用 `query_logistics(order_id="ORD20260623-001")`。
4. 后端执行该函数，向 `http://192.168.126.200/dev-api/logistics?orderId=...` 发送请求，获得物流信息 JSON。
5. 将工具返回结果再次传给模型，模型综合截图中的上下文，生成类似 “该订单的最新物流状态是【运输中】，预计 6 月 26 日送达……” 的自然语言答复。
6. 答复返回前端展示。

## 四、技术栈汇总推荐

| 层级             | 推荐技术                                     | 说明                               |
| :--------------- | :------------------------------------------- | :--------------------------------- |
| 前端             | React / Vue3, 支持粘贴图片                   | 可参考开源聊天项目如 LobeChat 定制 |
| 后端             | Python FastAPI + Celery (可选异步任务)       | 异步高性能，生态好                 |
| Agent 框架       | LangChain / LangGraph 或 自研轻量 ReAct 循环 | 若逻辑简单可手写，减少依赖         |
| 大模型服务       | vLLM + Qwen3-VL-Plus 等                      | 视觉+工具调用，本地部署            |
| 内部 API 访问    | httpx / aiohttp                              | 异步调用内部接口                   |
| 会话与存储       | Redis (对话缓存) + PostgreSQL (日志/审计)    | 按需选择                           |
| 向量数据库(可选) | Milvus                                       | 当存在静态文档需要检索时引入 RAG   |
| 部署             | Docker Compose                               | 各服务容器化，内网私有化部署       |

## 五、特别注意事项

1. **纯内网环境**：若集群无法访问公网，模型镜像、Python 包需提前下载并离线安装；模型权重文件需手动下载后挂载。
2. **截图解析的准确性**：多模态模型对中文界面、复杂表格的识别能力需验证，可提前用少量内部截图进行提示词优化或轻量微调。
3. **内部 API 文档化**：`dev-api` 应提供 OpenAPI/Swagger 规范，这样可以**自动生成工具描述**，大大减少手工封装工作量。
4. **性能优化**：对于高清截图，可在送入模型前压缩至 512~1024px，长边缩放，以降低推理延迟。
